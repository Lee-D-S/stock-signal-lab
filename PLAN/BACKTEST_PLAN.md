# 백테스팅 설계 계획

## 목적

정의된 조건 조합(28개 스크리닝 조건)을 과거 데이터에 적용하여  
실제로 수익이 났는지 검증하는 것.

> 어떤 조건이 유망한지 모르겠다면 → `FACTOR_RESEARCH_PLAN.md` 먼저 참고  
> AND 조합 대신 득표율 기반 선정을 원한다면 → `SCORING_PLAN.md` 참고

---

## 핵심 원칙

| 원칙 | 문제 | 대응 |
|------|------|------|
| 과적합 방지 | 과거에만 맞는 조합 선택 위험 | Train/Validation/Test 구간 분리 + Walk-forward |
| Look-ahead bias 제거 | 당시 알 수 없던 데이터 사용 | T일 조건 계산 → T+1일 시가 체결 |
| 거래비용 반영 | 수수료/슬리피지 미포함 시 결과 왜곡 | 왕복 약 0.24% 비용 적용 |
| 생존 편향 제거 | 현재 상장 종목만 테스트 시 결과 과대평가 | 당시 상장 종목 목록 기준으로 탐색 |

## 1. 데이터 구간 설계

### 기간 분리 (과적합 방지)

```
전체 기간: 2020-01-01 ~ 2024-12-31 (5년)

┌─────────────────┬────────────┬──────────┐
│   Train         │ Validation │   Test   │
│  2020 ~ 2022    │    2023    │   2024   │
│  (전략 탐색)     │ (파라미터   │ (최종     │
│                 │  튜닝)     │  검증)   │
└─────────────────┴────────────┴──────────┘
```

- **Test 구간은 전략 확정 후 딱 한 번만 실행**
  - 여러 번 보면 무의식적으로 Test에 맞게 과적합됨

### 생존 편향 제거

- KIS API는 현재 상장 종목만 반환 → 폐지 종목 누락
- **단기 대안**: 코스피200/코스닥150 구성 종목 히스토리 활용 (편입/편출 반영)
- **장기 대안**: DART `회사목록` API로 날짜별 당시 상장 종목 확보
- 구현 파일: `backtest/universe_loader.py`

## 2. 시뮬레이션 로직

### Look-ahead bias 방지 원칙

| 시점 | 허용 데이터 | 금지 데이터 |
|------|-----------|-----------|
| T일 조건 계산 | T-1일까지 OHLCV | T일 종가/거래량 |
| 매수 체결 | T+1일 시가 | T일 종가, T+1 종가 |
| 재무 조건 (DART) | 공시일 이후 수치만 | 공시 전 수치 |

```
T일:    종가 기준으로 28개 조건 스크리닝 (T-1일까지 데이터만 사용)
T+1일:  시가로 매수 체결 (당일 시가 + 슬리피지)
T+N일:  익절/손절/보유기간 만료 → 시가로 매도
```

### 거래비용 계산

```
수수료:   매수 0.015% + 매도 0.015%  (KIS 기본)
슬리피지: ±0.1% (시가 체결 시 불리한 방향 가정)

실효 비용: 편도 ≈ 0.115%
           왕복 ≈ 0.23%
```

### 포지션 관리 규칙

- 최대 동시 보유 N개 (기본 10)
- 동일 비중 배분 (Equal Weight)
- 신규 신호 발생 시 현재 포지션 수 확인 후 진입 여부 결정
- 동일 종목 중복 진입 금지

## 3. 성과 지표

| 지표 | 설명 | 기준선 |
|------|------|--------|
| **승률** | 수익 거래 / 전체 거래 | > 50% |
| **평균 수익률** | 거래당 평균 손익 | > 거래비용 (0.23%) |
| **손익비** | 평균 수익 / 평균 손실 | > 1.5 |
| **CAGR** | 연 복리 수익률 | > KOSPI 벤치마크 |
| **MDD** | 최대 낙폭 | < 20% |
| **샤프비율** | (수익률 - 무위험수익률) / 변동성 | > 1.0 |
| **거래 횟수** | 통계 신뢰도 확보 | > 30회 |

## 4. 과적합 방지 전략

### Walk-forward Analysis (핵심)

```
슬라이딩 윈도우로 전략의 시간 일관성 검증

[2020][2021][2022] → 전략 도출 → 2023 검증
      [2021][2022][2023] → 전략 도출 → 2024 검증
```

- 특정 구간에서만 잘 되는 전략 걸러냄

### 조건 수 제한 (Occam's Razor)

```
경험칙: 조건 수 ≤ 거래 횟수 / 10

예) 50회 거래 → 최대 5개 조건
    100회 거래 → 최대 10개 조건
```

- 조건이 많을수록 과거에 과적합될 위험 증가

### 복수 종목군 교차 검증

- 코스피에서 유효한 전략 → 코스닥에서도 유효한지 확인
- 시장 상황별 검증: 상승장(2020 하반기) / 하락장(2022) / 횡보(2023)

## 5. 구현 모듈 구조

```
scripts/
├── backtest/
│   ├── engine.py          # 날짜 순회, 신호 탐지, 체결 시뮬레이션
│   ├── portfolio.py       # 포지션 관리, 거래비용 계산, 수익률 추적
│   ├── metrics.py         # 승률, MDD, 샤프비율, CAGR 계산
│   ├── data_loader.py     # 날짜별 OHLCV 로딩 + 로컬 캐시 (SQLite or parquet)
│   ├── universe_loader.py # 날짜별 상장 종목 목록 (생존 편향 방지)
│   └── report.py          # 결과 출력, CSV 저장, 차트 (선택)
└── run_backtest.py        # 진입점 — 조건 조합 지정 후 실행
```

### 각 모듈 역할

**`data_loader.py`**
- 종목별 전체 OHLCV를 한 번 조회 후 메모리에서 날짜 순회
- 300종목 기준 API 호출 약 300번 → 약 2분 소요 (초기 실행 허용 범위)
- 조건 조합을 반복 탐색할 때 느리다 싶으면 로컬 캐시(SQLite/Parquet) 추가
- **캐시는 선택 사항 — 처음엔 없어도 동작함**

**`engine.py`**
- 날짜를 하루씩 순회하며 스크리닝 조건 적용
- Look-ahead bias 없도록 데이터 슬라이싱 보장
- 신호 발생 시 `portfolio.py`에 진입 요청

**`portfolio.py`**
- 포지션 추가/청산, 거래비용 차감
- 보유 기간, 익절/손절 조건 관리
- 일별 포트폴리오 가치 추적

**`metrics.py`**
- 거래 내역 → 성과 지표 계산
- 벤치마크(KOSPI) 대비 비교

**`report.py`**
- 콘솔 출력 + CSV 저장
- 선택: matplotlib으로 자산 곡선 차트

## 6. 실행 예시 (완성 후)

```bash
# PER 15 이하 + 장기 정배열 전략, 2020~2022 Train 구간
python scripts/run_backtest.py \
  --per-max 15 --ma-align 60,120,240 \
  --start 2020-01-01 --end 2022-12-31 \
  --hold-days 20 --max-positions 10

# ROE 10% 이상 + PBR 1 이하 + RSI 과매도
python scripts/run_backtest.py \
  --roe-min 10 --pbr-max 1 --rsi-max 40 \
  --start 2020-01-01 --end 2022-12-31 \
  --hold-days 10 --stop-loss -0.05 --take-profit 0.10
```

## 7. 구현 우선순위

```
1단계: data_loader  — 종목별 OHLCV 조회 (캐시 없이 시작 가능)
2단계: engine       — Look-ahead bias 없는 날짜 순회 루프
3단계: portfolio    — 거래비용 + 포지션 관리
4단계: metrics      — 성과 지표 계산
5단계: report       — 결과 출력 및 CSV 저장
6단계: walk-forward — 슬라이딩 윈도우 검증
7단계: data_loader 캐시  — 반복 탐색이 느릴 때 SQLite/Parquet 추가
8단계: universe_loader   — 생존 편향 제거 (DART 연동)
```

## 8. 제약 및 주의사항

- **KIS API 호출 방식**: 종목별 전체 기간 OHLCV를 한 번씩 조회 → 300종목 기준 약 2분. 조건 변경 시 재조회 필요하므로 반복 탐색이 잦아지면 캐시 추가 고려
- **DART 재무 조건**: 공시일 기준 반영 필요 (분기보고서 공시 후 데이터 사용)
- **거래 단위**: 한국 주식은 1주 단위 → 소수점 매수 불가, 수량 계산 시 floor 처리
- **배당락/액면분할**: OHLCV 수정주가 사용 여부 결정 필요 (KIS API: 수정주가 제공)
