# 주가 변동 원인 분석 프로젝트 종합 기획

## 1. 프로젝트 목적

KIS 시세/수급 데이터와 DART 공시/재무 데이터를 조합해, 종목별·분기별로 "주가가 왜 올랐고 왜 내렸는가"의 원인 후보를 정리하고, 여러 종목·여러 분기에서 반복되는 공통 패턴을 발견한다.

발견된 패턴은 즉시 확정 전략으로 사용하지 않는다. 가설로 보고 백테스트 또는 추가 관찰로 검증한 뒤에 실제 매매 기준으로 반영한다.

---

## 2. 핵심 질문

- 어떤 시그널 조합이 있을 때 상승 이후 가격이 유지되는가?
- 어떤 시그널 조합이 있을 때 상승 이후 가격이 되돌아오는가?
- 어떤 시그널 조합이 있을 때 하락 이후 추가 하락이 이어지는가?
- 어떤 시그널 조합이 있을 때 하락 이후 반등이 오는가?
- 이러한 패턴이 이벤트 이전에 관측 가능한 선행 시그널인가, 아니면 사후에만 확인 가능한 사후 설명인가?

### 4.1 통합 실행기

반복 작업은 `scripts/run_signal_research_pipeline.py`로 묶는다. MCP로 만들기 전 단계의 표준 실행 진입점이다.

| 모드 | 명령 | 용도 |
|------|------|------|
| 일별 후보 산출 | `rtk python -u scripts/run_signal_research_pipeline.py --mode daily` | 최신 거래일 기준 관심종목 후보 생성 |
| 일별 후보 + 수급 재조회 | `rtk python -u scripts/run_signal_research_pipeline.py --mode daily --recheck` | KIS 수급 가능 시간 이후 확정 후보 분리 |
| 가설/백테스트 재계산 | `rtk python -u scripts/run_signal_research_pipeline.py --mode backtest` | 이벤트 집계, 패턴 분석, 가설 리뷰, 실전 백테스트, 전략 초안 재생성 |
| 전체 재계산 | `rtk python -u scripts/run_signal_research_pipeline.py --mode full` | 보고서 재생성 없이 이벤트/백테스트 파이프라인 재계산 |
| 보고서 포함 전체 재계산 | `rtk python -u scripts/run_signal_research_pipeline.py --mode full --include-reports` | 540개 분기 보고서까지 포함해 전체 재생성 |
| 실행 계획 확인 | `rtk python -u scripts/run_signal_research_pipeline.py --mode backtest --dry-run` | 실제 실행 없이 단계 확인 |

운영 원칙:
- `--include-reports`는 오래 걸리므로 분석 기준이나 보고서 생성 코드가 바뀐 경우에만 사용한다.
- 장중에는 `--mode daily`로 후보만 만들고, KIS 수급 가능 시간 이후 `--mode daily --recheck`를 실행한다.
- 과거 이벤트/가설 결과를 다시 검증할 때는 `--mode backtest`를 기본으로 사용한다.

```
## 분석 전제
  - 분석 기간, 실제 거래일 범위, DART 연결 기준, 제외 항목, 결론 성격

## 기간 주가 요약
  - 시작/종료 종가, 기간 수익률, 기간 고/저점, 평균/최대 거래대금 (표)

## 주요 상승일
  - 상위 N개 상승 이벤트 (날짜, 종가, 등락률, 거래대금, 전후 외국인/기관, 주변 DART 공시) (표)

## 주요 하락일
  - 주요 하락 이벤트 (동일 구조) (표)

## 이벤트별 상승 원인 후보 판정
  └─ ### YYYY-MM-DD 상승 이벤트
      - 당일 등락률, 당일 거래대금
      - 후속 수익률: D+1, D+3, D+5, D+10
      - 후속 판정: 상승지속 / 상승유지 / 단기되돌림 / 상승실패
      - 시그널 태그 (복수 나열)
      - 원인 창 분류 (복수 나열)
      - 선행 시그널 여부: 있음 / 약함 / 없음
      - 멀티 윈도우 요약: (별도 섹션 아님, 이벤트 내부 bullet)
          - 선반영 창(D-20~D-1): 날짜 범위, 수익률, 평균 거래대금, 외국인/기관/개인 순매수, 공시 건수
          - 분기 선행 창(직전 분기): 동일 항목
          - 장기 배경 창(직전 6개월): 동일 항목
      - 창 분류 근거 (bullet)
      - 원인 후보 우선순위 (1~4순위)
      - #### 외부 자료 확인 필요 (조건부, 설명력 약한 이벤트에만 표시)

## 이벤트별 하락 원인 후보 판정
  └─ ### YYYY-MM-DD 하락 이벤트 (동일 구조, 후속 판정은 하락 계열)

## KIS 수급 요약
  - 분기 누적 외국인/기관/개인 순매수 (표)

## KIS 현재 참고 지표
  - 현재가, PER, PBR, EPS, BPS (표)

## KIS 공매도 요약
  - 공매도 거래대금 합계, 일평균, 최대 (표)

## DART 공시 요약
  - 분기 내 공시 목록 (날짜, 공시명) (표)

## DART 주요 재무 수치
  - 최근 5개 보고서 기준 매출액/영업이익/순이익/영업이익률/부채비율/ROE (표)

## DART 구조화 이벤트 조회 결과
  - 자기주식취득결정, 대량보유상황보고 등 주요 공시 상세

## 상승 원인 후보 우선순위
  - 분기 전체 기준 상승 원인 후보 1~4순위 요약

## 하락 원인 후보 우선순위
  - 분기 전체 기준 하락 원인 후보 1~4순위 요약

## 종합 판단
  - 이 분기 주가 변동의 전반적 해석
```

> - 멀티 윈도우 요약은 별도 `##` 섹션이 아니라 각 이벤트 `###` 블록 안에 bullet으로 들어간다.
> - `#### 외부 자료 확인 필요`는 조건부로 이벤트 내부에 표시되며, 별도 섹션이 아니다.

## 6. 이벤트 데이터 저장 전략

### 6.1 MD 파싱 대신 생성 시 동시 저장 권장

MD 텍스트를 나중에 파싱해 이벤트를 추출하는 방식은 취약하다. 보고서 문구가 조금만 바뀌거나 예외 케이스가 생기면 파싱이 깨진다.

권장 방식: 보고서 생성 시점(`build_period()`)에 MD와 함께 `events.jsonl`을 동시에 저장.
- MD: 사람이 읽는 산출물
- events.jsonl: 패턴 분석용 구조화 산출물

### 6.2 events.jsonl 스키마

각 줄은 이벤트 하나를 나타내는 JSON 객체.

| 필드 | 타입 | 설명 |
|------|------|------|
| `event_id` | str | 고유 식별자. 형식: `{ticker}_{quarter}_{date}_{direction}` (예: `005930_2025_Q3_2025-08-01_up`) |
| `ticker` | str | 종목코드 |
| `name` | str | 종목명 |
| `quarter` | str | 분기 코드 (예: `2023_Q2`) |
| `date` | str | 이벤트 날짜 (YYYY-MM-DD) |
| `direction` | str | `up` / `down` |
| `chg_pct` | float | 당일 등락률 (%) |
| `trade_amount` | float | 당일 거래대금 (원) |
| `amount_tag` | str | `거래대금급증` / `거래대금평균상회` / `거래대금약함` |
| `flow_tags` | list[str] | `외국인기관동반매수` / `외국인기관동반매도` / `외국인기관수급엇갈림` / `개인매수` / `개인매도` 등. CSV 변환 시 `\|`로 join |
| `dart_tag` | str | `DART공시동반` / `주변공시부재` |
| `window_types` | list[str] | `직접반응형` / `선반영형` / `누적배경형` / `외부충격형` / `설명부족형`. CSV 변환 시 `\|`로 join |
| `leading_signal` | str | `있음` / `약함` / `없음` |
| `followup` | str | `상승지속` / `상승유지` / `단기되돌림` / `상승실패` / `하락지속` / `하락유지` / `단기반등` / `하락후반등` |
| `needs_external_review` | bool | 내부 수치 설명력 부족 또는 방향 불일치 이벤트 여부 (`#### 외부 자료 확인 필요` 블록 존재 시 true) |
| `market_regime` | str | 해당 분기의 시장 국면 분류 (아래 섹션 참고) |

> `event_id`는 CSV, 보고서, 백테스트 결과를 나중에 join할 때 기준 키로 사용한다.
> `needs_external_review=true`인 이벤트는 Gemini/GPT 검색 또는 수동 뉴스 조사 대상 목록으로 별도 관리한다.

## 7. 패턴 분석 기획

### 7.1 5축 패턴 분석

아래 5개 축의 조합으로 이벤트를 그룹핑하고, 각 조합의 빈도와 후속 흐름 성공률을 집계한다.

| 축 | 분류 항목 |
|----|----------|
| 거래대금 강도 | 급증 / 평균상회 / 약함 |
| 수급 구조 | 외국인기관동반매수 / 외국인기관동반매도 / 수급엇갈림 / 개인중심 |
| 공시 유무 | 중요공시 동반 / 공시 없음 |
| 원인 창 유형 | 직접반응 / 선반영 / 누적배경 / 외부충격 / 설명부족 |
| 후속 흐름 | 상승지속/상승유지/단기되돌림/상승실패 (상승이벤트) / 하락지속/하락유지/단기반등/하락후반등 (하락이벤트) |

### 7.2 패턴 유효 기준

패턴이 의미 있으려면 아래 조건을 모두 충족해야 한다.

- 동일 조합이 **10회 이상** 등장
- 특정 후속 흐름 비율이 **65% 이상**
- 특정 종목에만 집중되지 않고 **3개 이상 종목**에서 등장
- **시장 국면을 나눠서도 성립하는지** 별도 확인 (아래 참고)

이 기준을 통과한 패턴만 가설로 등록한다.

### 7.3 시장 국면 분리

같은 시그널 조합이라도 시장 국면에 따라 후속 흐름이 달라질 수 있다. 전 기간을 합산한 패턴이 특정 국면에서만 작동하는 패턴을 가릴 수 있고, 반대로 국면마다 다른 방향으로 작동하는 패턴이 합산에서 희석될 수 있다.

분기별로 아래 국면 중 하나를 `market_regime`으로 분류하고, 패턴 분석 시 국면별로 분리해 검증한다.

| 국면 | 해당 기간 (잠정) | 특징 |
|------|----------------|------|
| 유동성/저금리 후반장 | 2021_Q2 ~ 2021_Q4 | 코로나 이후 유동성 장세 후반, 인플레이션 초입, 전반적 상승 기조 |
| 금리 인상/긴축장 | 2022_Q1 ~ 2023_Q1 | 고금리, 성장주 디레이팅, 전반적 하락 압력 |
| 반도체 반등장 | 2023_Q2 ~ 2023_Q4 | 반도체 업황 회복 기대, 외국인 복귀 |
| AI/전력기기 테마장 | 2024_Q1 ~ 2025_Q2 | AI 인프라/방산/전력기기 테마 주도, 종목 쏠림 심화 |
| 변동성 장세 | 2025_Q3 ~ 2026_Q1 | 거시 불확실성, 분기별 방향 불명확 |

> 이 분류는 잠정적이며, 분석 진행 중 조정 가능.

### 7.4 사후 설명 vs 선행 시그널 구분

| 구분 | 정의 | 활용 |
|------|------|------|
| 사후 설명 후보 | 이벤트 발생 이후에 확인 가능한 원인 후보 | 패턴 이해, 사후 분석 |
| 선행 시그널 후보 | **이벤트 발생 전날까지** 이미 관측 가능했던 데이터에서 나온 후보 | 실제 매매 시그널 후보 |

**선행 시그널 판정 원칙:**
- D-20~D-1의 외국인/기관 누적 순매수 방향: 선행 가능
- 직전 분기 실적 개선 여부: 선행 가능
- 직전 6개월 반복 공시, 장기 수급 누적: 선행 가능
- D+1, D+3, D+5 후속 가격 유지 여부: **선행 불가** (결과 라벨, 진입 시그널 아님)
- 이벤트 당일 공시 내용: 장 시작 전 공시면 선행 가능, 장 중/마감 후 공시면 사후
- D-5~D+5 창의 수급: 이벤트 전날까지의 값만 선행으로 분류

`leading_signal` 필드가 `약함`인 경우: 선행 데이터가 존재하긴 하나 방향성이 약하거나 단일 지표에만 의존하는 경우.

### 7.6 후보 리뷰 및 1차 프록시 검증 결과

국면별 후보 6개를 실제 이벤트 86건으로 다시 펼쳐 `가설_이벤트_검토.md`에 정리했다. 각 후보는 `진입 후보`, `반등 관찰 후보`, `추격매수 회피 후보`로 분류했다.

`가설_대리_백테스트.md`는 이벤트 당일 종가 기준 `D+5/D+10` 후속 수익률을 이용한 1차 프록시 검증이다. 실제 매매 검증은 아니며, 다음 거래일 시가/종가 진입과 거래비용은 아직 반영하지 않았다.

| 후보 | 해석 | D+5 평균 점수 | D+5 적중률 | D+10 평균 점수 | D+10 적중률 | 주의 |
|------|------|--------------:|-----------:|---------------:|------------:|------|
| H01 | 진입 후보 | +16.20% | 92.86% | +22.86% | 85.71% | 외부 검토 필요 0건 |

| H04 | 추격매수 회피 후보 | +6.00% | 83.33% | +7.62% | 70.00% | 외부 검토 필요 12건 |

## 8. 외부 요인 검색 (Gemini) 계획

내부 수치(KIS/DART)만으로 설명력이 약한 이벤트에 대해 외부 뉴스/정책/업황을 Gemini API로 검색한다.

### 트리거 조건

아래 조건 중 하나라도 해당하면 외부 검색 실행:

- **조건 A (신호 약함)**: 거래대금 태그가 `거래대금약함`이고 수급도 `수급엇갈림`이면서 주변 공시 없음
- **조건 B (방향 불일치)**: KIS/DART 수치상 방향이 상승 우세인데 실제 주가는 하락하거나, 반대의 경우

events.jsonl의 `needs_external_review=true` 이벤트가 외부 검색 대상과 일치한다.

### 구현 상태

- 코드: `scripts/tmp_quarterly_stock_analysis.py`의 `query_gemini_external()` 함수 구현 완료
- 현재 상태: **주석 처리 (비활성화)**
- 비활성화 이유: Gemini 무료 플랜 RPD=20 (gemini-2.5-flash 기준)으로 27개 종목 배치에 부족
- 재활성화 조건: Gemini API 유료 플랜 전환 또는 GPT API 대체 사용

### 출력 형식

```
요인: [외부 요인 제목]
근거 날짜: YYYY-MM-DD
출처명: [매체명]
URL: [기사 URL]
관련성: [상/중/하]
신뢰도: [상/중/하]
```

## 11. 패턴 발견 후 사용 원칙

발견된 패턴은 가설이다. 아래 절차를 거친 뒤에만 실제 매매 판단에 반영한다.

1. **패턴 등록**: 유효 기준(10회+, 65%+, 3종목+) 충족 확인
2. **국면 검증**: 전 기간 합산 외에 국면별로 분리해서도 패턴이 유지되는지 확인
3. **선행 여부 확인**: 이벤트 전날까지 관측 가능한 시그널인지 검토 (D+3, D+5 후속 흐름은 결과 라벨이지 진입 시그널이 아님)
4. **백테스트**: 기존 `run_backtest.py` 또는 별도 스크립트로 검증
5. **추가 관찰**: 실시간 환경에서 신규 이벤트 발생 시 패턴 일치 여부 관찰
6. **전략 반영**: 1~5 통과 후 스크리너 조건 또는 전략 플러그인으로 통합
