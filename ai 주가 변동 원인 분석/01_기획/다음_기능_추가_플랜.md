# 다음 기능 추가 플랜

## 목적

현재 일일 자동화는 정상 동작한다. 다음 단계는 자동화 범위를 무리하게 넓히는 것이 아니라, 관찰 품질과 운영 안정성을 높이는 기능을 작게 추가하는 것이다.

우선순위는 다음 4개다.

1. 신규 조건 별도 관찰 모드
2. 관찰 결과 집계 리포트
3. GitHub Actions 실패 텔레그램 알림
4. DART 공시 본문/수치 기반 해석

## 1. 신규 조건 별도 관찰 모드

### 목적

새로 발견된 조건을 기존 active 조건 `H01/H02/H03/H04/H06`과 섞지 않고 별도 검증한다.

우선 대상:

- `NEW-20260501-H05`
- `NEW-20260501-H07`

### 현재 문제

신규 조건을 바로 active에 넣으면 기존 조건 검증 흐름이 흐려진다. 같은 관찰 로그에 섞이면 H01/H02의 실제 성과와 신규 조건의 실험 성과를 분리하기 어렵다.

### 구현 방향

별도 strategy CSV와 별도 관찰 로그를 둔다.

입력:

- `09_조건스냅샷/2026-05-01/`
- 신규 조건 검토 결과
- 거래대금 top30 유니버스

출력:

- `07_전략신호/신규조건_관심종목_시그널_후보.csv`
- `07_전략신호/신규조건_관심종목_시그널_후보.md`
- `08_관찰기록/신규조건_관찰_로그.csv`
- `08_관찰기록/신규조건_관찰_로그.md`

작업 단위:

1. 신규 조건 전용 CSV 생성
2. `tmp_generate_watchlist_signals.py`의 `--strategy-csv` 옵션으로 신규 조건만 스캔
3. 결과 파일명을 active 조건 결과와 분리
4. 신규 조건 전용 관찰 로그 추가 스크립트 작성
5. D+ 추적도 신규 조건 로그에 별도 적용

주의:

- 기존 `관찰_로그.md`에 신규 조건을 섞지 않는다.
- 기존 H 번호를 재사용하지 않는다.
- 최소 20건 이상 쌓이기 전에는 active 승격하지 않는다.

## 2. 관찰 결과 집계 리포트

### 목적

관찰 로그에 쌓인 D+1/D+5/D+10/D+20 결과를 조건별로 자동 집계한다.

### 현재 문제

개별 후보의 결과는 `관찰_로그.md`에서 볼 수 있지만, 조건별 평균 성과와 성공/실패 비율을 한눈에 보기 어렵다.

### 구현 방향

관찰 로그를 읽어서 조건별 요약 파일을 만든다.

입력:

- `08_관찰기록/관찰_로그(이상).csv`

출력:

- `08_관찰기록/관찰_성과_요약.csv`
- `08_관찰기록/관찰_성과_요약.md`

집계 항목:

- `hypothesis_id`
- `use_type`
- 표본 수
- D+1 종가 평균 수익률
- D+5 평균 수익률
- D+10 평균 수익률
- D+20 평균 수익률
- 성공/실패/중립 라벨 비율
- 표본 부족 여부

작업 단위:

1. `scripts/run_observation_performance_summary.py` 생성
2. `run_observation_tracking_update.py` 이후 실행
3. GitHub Actions daily job에 추가
4. 일일 운영 요약에서 성과 요약 파일 링크 또는 핵심 수치 표시

주의:

- 표본 20건 미만 조건은 결론 대신 `표본 부족`으로 표시한다.
- 회피 조건과 매수/반등 조건은 성공 기준이 다르므로 라벨 해석을 분리한다.

## 3. GitHub Actions 실패 텔레그램 알림

### 목적

자동화 실패를 GitHub Actions 화면에 들어가지 않아도 알 수 있게 한다.

### 현재 문제

workflow가 실패하면 사용자가 직접 Actions 탭을 확인해야 한다. KIS/DART/API 오류가 발생해도 늦게 알 수 있다.

### 구현 방향

각 job 마지막에 실패 시 텔레그램 알림 step을 추가한다.

입력:

- GitHub Actions job 상태
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`

출력:

- 텔레그램 실패 메시지

메시지 예:

```text
[auto-invest] GitHub Actions 실패
workflow: 일별 자동 루틴
job: daily-research-pipeline
branch: main
run: <GitHub Actions URL>
```

작업 단위:

1. `.github/workflows/daily_auto.yml`에 실패 알림 step 추가
2. `if: failure()` 조건 사용
3. 오전 DART job과 오후 daily job에 각각 적용
4. 텔레그램 secret 없으면 조용히 skip

주의:

- 성공 알림은 처음에는 추가하지 않는다. 알림 피로가 커진다.
- 실패 URL을 포함해서 바로 로그로 이동할 수 있게 한다.

## 4. DART 공시 본문/수치 기반 해석

### 목적

공시 제목만으로 판단하기 어려운 공시를 실제 수치와 맥락으로 해석한다.

### 현재 상태

오전 DART 공시 확인은 제목 기반으로 1차 분류한다.

분류:

- `호재성/보강`
- `악재성/리스크`
- `중립/확인`
- `본문확인필요`
- `미분류/확인필요`

실적, 매출/손익구조, 유상증자, 전환사채, 최대주주, 합병/분할, 투자판단 관련 공시는 제목만으로 방향을 확정하지 않는다.

### 구현 방향

`본문확인필요` 공시에 대해 DART 상세 API 또는 공시 원문 링크를 조회하고 핵심 수치를 추출한다.

우선 처리할 공시 유형:

1. 실적/잠정실적
2. 매출액 또는 손익구조 변경
3. 단일판매/공급계약
4. 유상증자/전환사채

출력:

- 관찰 로그 `review_note`에 근거 포함
- `관찰_로그.md` 오전 DART 공시 섹션에 해석 추가

해석 예:

```text
오전 DART 공시: 영업(잠정)실적
[본문확인필요 -> 실적개선: 매출 +18%, 영업이익 +42%; 반등 후보 보강]
```

작업 단위:

1. DART 상세 조회 함수 추가
2. 공시 유형별 parser 추가
3. 실적/계약/증자 유형부터 수치 추출
4. 수치 추출 실패 시 `본문확인필요 유지`
5. 관찰 로그와 markdown에 근거 포함

주의:

- 본문 파싱은 공시 유형별 포맷 차이가 커서 한 번에 완성하지 않는다.
- 처음에는 실적/공급계약 두 유형만 구현해도 충분하다.
- 수치 추출이 불확실하면 자동 호재/악재로 확정하지 않는다.

## 권장 구현 순서

1. 관찰 결과 집계 리포트
2. GitHub Actions 실패 텔레그램 알림
3. 신규 조건 별도 관찰 모드
4. DART 공시 본문/수치 기반 해석

이유:

- 1번은 지금 쌓이기 시작한 관찰 데이터를 바로 활용할 수 있다.
- 2번은 운영 안정성을 높인다.
- 3번은 신규 조건 검증을 시작할 수 있지만, 별도 파일 체계가 필요하다.
- 4번은 효과는 크지만 공시 유형별 파싱 난도가 높아 단계적으로 해야 한다.
