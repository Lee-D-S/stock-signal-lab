# 일일 운영 자동화 정리

## 현재 결론

이 프로젝트의 일일 운영은 이제 GitHub Actions와 로컬 자동 pull까지 연결되어 있다.

하루 흐름은 다음과 같다.

```text
08:50 GitHub Actions: 관찰 종목 DART 공시 확인
09:05 로컬 PC: GitHub 결과 자동 pull
16:10 GitHub Actions: 거래대금 top30 갱신 -> 후보 탐지 -> 수급 재확인 -> 관찰 로그/D+ 추적 -> 일일 요약
16:35 로컬 PC: GitHub 결과 자동 pull
```

GitHub Actions가 직접 갱신하는 곳은 GitHub 저장소다. 로컬 `C:\projects\auto-invest`는 Windows 작업 스케줄러의 `AutoInvest Local Pull` 작업이 평일 09:05, 16:35에 `git fetch`와 `fast-forward merge`로 최신 결과를 가져온다.

## 하루 운영 흐름

### 1. 평일 08:50 KST - 오전 DART 공시 확인

GitHub Actions job:

- `오전 DART 공시 확인 (08:50 KST)`

실행 스크립트:

```powershell
rtk python -u scripts/auto_morning_dart_check.py
```

입력 데이터:

- `08_관찰기록/관찰_로그(이상).csv`
- `data/dart_corp_codes.json`

처리 내용:

- 관찰 로그에서 아직 `result_label`이 비어 있는 종목을 찾는다.
- 각 종목의 DART 법인코드를 확인한다.
- 최근 3일 범위의 DART 공시를 조회한다.
- 신규 공시가 있으면 텔레그램으로 보낸다.
- 신규 공시가 있으면 관찰 로그의 `review_note`와 `관찰_로그.md`에 오전 공시 메모를 남긴다.
- 공시 제목을 기준으로 `호재성/보강`, `악재성/리스크`, `중립/확인`, `본문확인필요`, `미분류/확인필요`로 1차 분류한다.
- 관찰 후보 유형과 결합해 `기존 후보 보강`, `반등 후보 보강`, `기존 후보 약화/리스크`, `회피 조건 보강/약화 가능` 같은 해석 문구를 남긴다.
- 텔레그램 설정이 없으면 Actions 로그에 출력한다.

얻는 정보:

- 관찰 중인 후보에 새 공시가 붙었는지 알 수 있다.
- 신호 이후 나온 공시가 주가 흐름을 바꿀 수 있는지 확인할 수 있다.
- D+1/D+5/D+10/D+20 결과를 나중에 해석할 때 공시 영향을 따로 구분할 수 있다.
- 단순 공시 존재 여부가 아니라, 제목만으로도 비교적 방향성이 있는 공시와 본문 확인이 필요한 공시를 구분할 수 있다.
- 신규 공시가 없으면 “관찰 종목 신규 공시 없음”으로 종료한다.

자동 해석의 한계:

- 현재 공시 본문 전문을 읽지는 않는다.
- 제목 키워드 기반 1차 분류이므로 최종 판단은 관찰 로그와 실제 주가 흐름을 함께 봐야 한다.
- 실적, 매출/손익구조 변경, 유상증자, 전환사채, 최대주주 변경, 합병/분할, 투자판단 관련 공시는 제목만으로 방향을 확정하지 않고 `본문확인필요`로 남긴다.
- 미분류 공시는 `제목만으로 판단 불가`로 남긴다.

주의:

- 이 job은 후보를 새로 찾는 작업이 아니다.
- 관찰 중인 종목의 공시만 확인한다.
- DART 조회 실패는 종목별로 넘기고 전체 job을 죽이지 않는다.

### 2. 평일 09:05 KST - 로컬 오전 자동 pull

Windows 작업 스케줄러:

- `AutoInvest Local Pull`

실행 스크립트:

```powershell
rtk powershell -NoProfile -ExecutionPolicy Bypass -File scripts\local_auto_pull.ps1
```

처리 내용:

- 로컬 작업 트리가 깨끗한지 확인한다.
- `origin/main`을 fetch한다.
- 로컬 브랜치가 원격보다 뒤처진 정상 상태일 때만 fast-forward로 갱신한다.
- 로컬 변경사항이 있으면 pull을 건너뛴다.

로그:

- `.local/local_auto_pull.log`

의미:

- 오전 DART 공시 확인 결과가 GitHub에 반영된 경우 로컬에도 내려온다.
- 로컬에서 수정 중인 파일이 있으면 자동 병합하지 않는다.

#### 3-7. 결과 커밋 및 푸시

GitHub Actions는 `ai 주가 변동 원인 분석/` 아래 파일이 바뀐 경우만 자동 commit/push 한다.

변경 없음:

```text
변경 없음
```

변경 있음:

```text
auto: 일일 후보 및 관찰 업데이트 YYYY-MM-DD
```

의미:

- 변경 없음이면 오늘 새로 저장할 결과가 없었다는 뜻이다.
- 변경 있음이면 GitHub 저장소에 새 결과가 반영된다.

## 어떤 정보를 얻고 무엇을 판단하는가

### 거래대금 top30

알 수 있는 것:

- 오늘 시장에서 자금이 몰린 종목

- 분석 보고서가 필요한 신규 기업

판단:

- 신규 기업 보고서 생성 필요 여부
- 유니버스가 특정 섹터에 쏠리는지
- 기존 active 조건이 현재 시장에서 후보를 낼 수 있는지

### 전략 후보

알 수 있는 것:

- 기존 검증 조건과 유사한 가격/수급/공시 조합이 오늘 발생했는지
- 후보가 매수형인지, 반등 감시형인지, 회피형인지
- 백테스트 평균 점수와 hit rate가 어떤지

판단:

- 관찰 대상 추가 여부
- 추격 금지 또는 반등 확인 필요 여부
- 다음 거래일 주의해서 볼 종목

### 수급 재조회

알 수 있는 것:

- 초기 후보의 수급 조건이 실제로 맞는지
- 외국인/기관 동반 매수인지, 동반 매도인지, 엇갈림인지

판단:

- 관찰 로그에 넣을지 말지
- 신호 품질이 충분한지
- 수급 불일치로 탈락해야 하는지

### 관찰 로그

알 수 있는 것:

- 신호 이후 실제 수익률
- 조건별 성공/실패 표본
- D+1, D+5, D+10, D+20 구간별 반응

판단:

- H01/H02/H03/H04/H06 조건을 계속 유지할지
- 특정 조건을 약화 또는 제외할지
- 신규 조건을 active로 승격할 만큼 근거가 쌓였는지

### 오전 DART 공시

알 수 있는 것:

- 관찰 중인 종목에 신호 이후 새 공시가 나왔는지
- 기존 가격/수급 신호에 공시 이벤트가 추가됐는지

판단:

- 관찰 메모 보강 필요 여부
- 예상과 다른 주가 흐름의 사후 원인 후보
- 장중 주의 종목 여부

## GitHub Actions 확인 방법

GitHub 저장소에서:

```text
Actions -> 일별 자동 루틴 -> 최근 run -> daily-research-pipeline
```

볼 것:

| Step | 확인 내용 |
| --- | --- |
| `실행 커밋 및 DART 캐시 확인` | 최신 커밋인지, DART 캐시가 있는지 |
| `일별 후보 탐지 및 관찰 로그 추가` | `universe_count`, `candidates`, `confirmed` |
| `관찰 로그 D+ 추적 업데이트` | `observations`, `updated` |
| `관찰 성과 요약 생성` | `observations`, `summary_rows` |
| `일일 운영 요약 생성` | 요약 파일 생성 여부 |
| `결과 커밋 및 푸시` | 변경 없음 또는 자동 커밋 여부 |
| `성공 텔레그램 알림` | 성공 시 run URL이 포함된 텔레그램 메시지 |
| `실패 텔레그램 알림` | 실패 시 run URL이 포함된 텔레그램 메시지 |

텔레그램 알림 테스트:

1. GitHub 저장소의 `Actions` 탭으로 이동한다.
2. `일별 자동 루틴` workflow를 선택한다.
3. `Run workflow`를 누른다.
4. `job` 입력값을 `telegram-test`로 넣고 실행한다.
5. 텔레그램에 `[auto-invest] GitHub Actions 테스트` 메시지가 오면 정상이다.

정상 로그 예:

```text
DART 법인코드 캐시 사용: ... (3,960개)
universe_count=30
candidates=1
rechecked=1
confirmed 1
added=0
```

해석:

- `candidates=1`: 조건 후보 1개 발견
- `confirmed 1`: 수급 확인까지 통과
- `added=0`: 이미 관찰 로그에 있어 중복 추가하지 않음

## 로컬 자동 pull 확인 방법

작업 스케줄러:

- 이름: `AutoInvest Local Pull`
- 시간: 평일 09:05, 16:35

로그:

```text
C:\projects\auto-invest\.local\local_auto_pull.log
```

정상 예:

```text
up-to-date: <commit>
updated: <old> -> <new>
```

건너뛰는 예:

```text
skip: working tree has local changes
skip: local branch diverged from origin/main
```

건너뛴 경우는 자동 병합하지 않는 것이 맞다. 로컬 수정과 GitHub Actions 결과가 충돌할 수 있기 때문이다.

## 수동 실행 명령

GitHub Actions를 쓰지 않고 로컬에서 직접 돌릴 때:

```powershell
rtk python -u scripts/auto_morning_dart_check.py
rtk python -u scripts/run_signal_research_pipeline.py --mode daily --refresh-universe --recheck
rtk python -u scripts/run_observation_tracking_update.py
rtk python -u scripts/run_daily_research_summary.py
```

특정 기준일로 재실행:

```powershell
rtk python -u scripts/run_signal_research_pipeline.py --mode daily --date YYYY-MM-DD --refresh-universe --recheck
rtk python -u scripts/run_observation_tracking_update.py --as-of YYYY-MM-DD
rtk python -u scripts/run_daily_research_summary.py --date YYYY-MM-DD
```

로컬 자동 pull 재등록:

```powershell
rtk powershell -NoProfile -ExecutionPolicy Bypass -File scripts\register_local_auto_pull.ps1
```
