# 일일 운영 자동화 정리

## 목적

매일 같은 지시를 반복하지 않도록, 거래대금 상위 유니버스 갱신부터 신규 기업 추가, 전략 신호 스캔, 관찰 로그 기록까지의 운영 순서를 문서화한다.

이 문서는 구현 지시가 아니라 운영 기준 문서다. 실제 자동화 스크립트 수정은 별도 작업으로 진행한다.

## 기본 원칙

- 매일 기준 유니버스는 고정 27개가 아니라 최신 거래대금 상위 종목을 기준으로 갱신한다.
- 기존 분석 폴더에 없는 기업이 거래대금 상위 유니버스에 새로 들어오면 기업 분석 폴더와 기본 보고서 생성 대상으로 추가한다.
- 전략 신호는 검증된 조건 세트(H01/H02/H03/H04/H06)를 먼저 적용한다.
- 새 패턴 분석 결과는 기존 조건 번호에 바로 섞지 않는다. 신규 분석 결과는 `09_조건스냅샷/YYYY-MM-DD/`에 보관하고, active 조건으로 승격할 때만 명시적으로 반영한다.
- 신호는 매수 지시가 아니라 관찰 샘플이다. 확정 후보만 관찰 로그에 추가한다.
- D+1, D+5, D+10, D+20 결과를 누적해서 조건의 실제 성과를 판단한다.
- 최근 하루 결과만 보고 조건을 바꾸지 않는다. 최소 20건 이상 누적한 뒤 조건 추가/제외를 검토한다.

## 매일 갱신할 범주

### 1. 거래대금 상위 유니버스

매 거래일 장 마감 후 최신 거래대금 기준으로 국내주식 상위 종목을 다시 산출한다.

운영 기준:

- 기준일: 최신 거래일
- 정렬 기준: 당일 거래대금 내림차순
- 기본 개수: 상위 30개
- 제외 후보: 거래정지, 관리종목, 우선주, 스팩, ETF/ETN, 신규상장 직후 데이터 부족 종목
- 포함 후보: 기존 27개에 없더라도 거래대금 상위권에 새로 진입한 보통주

산출물 예시:

- `07_전략신호/거래대금_상위_유니버스.csv`
- `07_전략신호/거래대금_상위_유니버스.md`

필수 컬럼:

| 컬럼 | 의미 |
| --- | --- |
| 기준일 | 유니버스 산출 기준일 |
| 순위 | 거래대금 순위 |
| ticker | 종목코드 |
| name | 종목명 |
| close | 종가 |
| chg_pct | 등락률 |
| trade_amount | 거래대금 |
| volume | 거래량 |
| universe_status | 기존/신규/제외 |
| company_folder_exists | 기업 분석 폴더 존재 여부 |
| report_status | 보고서 존재/생성필요/보류 |

### 2. 신규 기업 추가

거래대금 상위 30개 안에 들어왔지만 `00_기업별분석/{종목명}` 폴더가 없으면 신규 기업으로 분류한다.

신규 기업 처리 기준:

- 기업 폴더 생성 대상에 추가
- DART 기업코드 매핑 확인
- 과거 OHLCV 캐시 생성 대상에 추가
- 분기별 원인 후보 분석 보고서 생성 대상에 추가
- 전략 신호 스캔 유니버스에 포함

단, 아래 경우는 보류한다.

- 상장 이력이 짧아 분기별 분석 샘플이 부족한 경우
- DART 기업코드 매핑이 실패한 경우
- 우선주/스팩/ETF/ETN 등 기존 분석 목적과 맞지 않는 경우
- 일시적 테마 급등으로 거래대금만 급증했고 기본 데이터가 부족한 경우

보류 종목도 별도 목록에 남긴다.

산출물 예시:

- `01_기획/신규_기업_추가_대상.md`
- `03_원천데이터/신규_기업_데이터_수집_상태.csv`

## 매일 운영 루틴

### 장 시작 전

목적은 전일 확정 후보와 오늘 추적 대상만 확인하는 것이다.

해야 할 일:

- 관찰 로그에서 D+1/D+5/D+10/D+20 도래 종목 확인
- 기존 관찰 종목의 전일 종가와 주요 공시 확인
- 오늘 자동 실행할 기준일이 최신 거래일인지 확인

이 단계에서는 새 전략 조건을 만들지 않는다.

### 장중

목적은 관찰 중인 종목의 과열/약세 여부만 확인하는 것이다.

H01 매수 후보:

- 다음 거래일 시가가 신호일 종가 대비 과도하게 높으면 추격 보류
- 장초반 급락이면 당일 진입 보류
- 종가가 신호일 종가 아래로 밀리면 약화 후보로 분류

H02/H03 반등 감시 후보:

- 신호일 종가 회복 여부 확인
- 시가 회복, 장중 평균가 회복 여부 확인
- 회복 없이 추가 하락하면 반등 실패 가능성으로 분류

H04/H06 추격 회피 후보:

- 추가 상승 지속 여부만 관찰
- 진입 후보가 아니라 과열/리스크 검증 샘플로 유지

### 장 마감 후

가장 중요한 일일 작업 구간이다.

순서:

1. 최신 거래대금 상위 종목 산출
2. 상위 30개 유니버스 갱신
3. 신규 진입 기업 확인
4. 신규 기업 폴더/보고서 생성 필요 목록 작성
5. 30개 유니버스에 전략 조건 적용
6. KIS 수급 재조회로 확정 후보와 탈락 후보 분리
7. 확정 후보만 관찰 로그에 추가
8. 기존 관찰 후보의 D+1/D+5/D+10/D+20 결과 기록
9. 당일 운영 요약 작성

주의:

- 일별 후보 탐지는 기본적으로 active 조건 파일 `07_전략신호/전략_조건_초안.csv`만 사용한다.
- 신규 조건 스냅샷을 실험하려면 별도 `--strategy-csv`를 지정하되, 관찰 로그에는 기존 active 후보와 섞어 기록하지 않는다.
- 패턴 재분석은 일일 후보 탐지와 별개 작업이다. 재분석 결과는 스냅샷으로 보관하고, 기존 조건 검증 흐름을 덮어쓰지 않는다.

## 전략 신호 스캔 범위

우선 적용 범위는 다음으로 제한한다.

- 최신 거래대금 상위 30개 보통주
- 기업 분석 폴더와 데이터 수집 상태가 준비된 종목
- DART 공시와 KIS 시세/수급 데이터 조회가 가능한 종목

적용 조건 세트:

| 가설 | 용도 | 핵심 조건 |
| --- | --- | --- |
| H01 | 매수 후보 | 변동성 장세 + 상승 + 거래대금약함 + DART공시 + 외국인기관동반매수 |
| H02 | 반등 감시 후보 | 변동성 장세 + 하락 + 거래대금약함 + DART공시 + 외국인기관동반매수 |
| H03 | 단기 반등 감시 후보 | AI/전력기기 테마장 + 하락 + 거래대금평균상회 + 수급엇갈림 + 주변공시부재 |
| H04 | 참고 회피 후보 | 유동성/저금리 장세 + 상승 + 거래대금급증 + 외국인기관동반매도 + DART공시 |
| H06 | 추격매수 회피 후보 | 반도체 반등장 + 상승 + 거래대금급증 + 외국인기관동반매도 + DART공시 |

현재 관찰 로그에 들어간 최근 확정 후보:

| 신호일 | 종목 | 통과 조건 | 상태 |
| --- | --- | --- | --- |
| 2026-04-29 | 한미반도체 | H02 | D+ 추적 중 |
| 2026-04-29 | 두산에너빌리티 | H01 | D+ 추적 중 |
| 2026-04-30 | 삼성전자 | H02 | D+ 추적 중 |

## 신규 조건 스냅샷 운영

신규 기업 보고서/events를 추가한 뒤 패턴을 재분석하면, 새 후보 번호가 다시 붙는다. 이 번호는 기존 active 조건 번호와 같은 의미가 아니므로 날짜를 붙여 구분한다.

2026-05-01 스냅샷 기준:

| 구분 | 개수 | 처리 |
| --- | ---: | --- |
| 오늘 스냅샷에서 발견된 조건 후보 | 9개 | `09_조건스냅샷/2026-05-01/` 보관 |
| 기존 active와 동일한 조건 | 3개 | 기존 active ID 유지, 성능 업데이트 후보로만 참고 |
| 기존 active에는 없던 신규 후보 | 6개 | 검증 대기 |
| 기존 active였지만 이번 스냅샷에서 빠진 조건 | 2개 | 삭제하지 않고 보류/관찰 유지 |

신규 6개 검토 결과:

| stable 후보 ID | 판단 | 비고 |
| --- | --- | --- |
| NEW-20260501-H05 | 우선 관찰 | 변동성 장세 상승, 거래대금약함, 외국인기관동반매수, 주변공시부재 |
| NEW-20260501-H07 | 우선 관찰 | AI/전력기기 테마장 하락, 거래대금약함, 외국인기관동반매수, 주변공시부재 |
| NEW-20260501-H01 | 소표본 보류 | 표본 부족 |
| NEW-20260501-H02 | 소표본 보류 | 표본 부족 |
| NEW-20260501-H04 | 보류 | 평균 점수 우위 약함 |
| NEW-20260501-H09 | 승격 제외 | 회피 조건 점수 부적합 |

운영 기준:

- active 조건에 즉시 추가한 신규 조건은 없다.
- 우선 관찰 2개도 기존 `H01~H06` 번호 체계에 섞지 않는다.
- 신규 조건은 최소 20건 이상 관찰 표본이 쌓일 때까지 스냅샷 검토 대상으로 유지한다.
- 승격 시에는 기존 H 번호를 재사용하지 말고 별도 stable ID를 부여한 뒤 active 조건 파일에 명시적으로 추가한다.

## 관찰 로그 추가 기준

관찰 로그에 추가하는 대상:

- 전략 조건을 통과한 후보
- 수급 재조회까지 통과한 확정 후보
- 신규 기업이라도 데이터가 충분하고 확정 후보로 판정된 종목

관찰 로그에 추가하지 않는 대상:

- `flow_check_required` 상태로 수급 확인이 끝나지 않은 후보
- `rejected_flow_mismatch` 상태로 수급 조건이 불일치한 후보
- 데이터 조회 오류로 신호 품질을 확인할 수 없는 후보
- 우선주/ETF/ETN/스팩 등 분석 범위 밖 종목

필수 기록:

| 항목 | 설명 |
| --- | --- |
| signal_date | 신호 발생일 |
| ticker/name | 종목코드/종목명 |
| hypothesis_id | H01/H02/H03/H04/H06 |
| use_type | 매수 후보/반등 감시 후보/회피 후보 |
| event_close | 신호일 종가 |
| flow_category_confirmed | 확정 수급 분류 |
| next_trading_day | 다음 거래일 |
| next_open/next_close | 다음 거래일 시가/종가 |
| D+5/D+10/D+20 | 추적 종가와 수익률 |
| result_label | 중간/최종 결과 라벨 |
| review_note | 해석 메모 |

## 실행 명령 초안

현재 기준 실제 운영 명령은 아래와 같다.

```powershell
# 1. 최신 거래대금 상위 유니버스 갱신
rtk python -u scripts/run_daily_universe_refresh.py --top 30

# 2. 신규/기존 누락 기업 보고서 생성
rtk python -u scripts/run_new_company_reports.py --include-existing-missing

# 3. 최신 유니버스 기준 active 전략 신호 스캔 및 수급 재조회
rtk python -u scripts/run_signal_research_pipeline.py --mode daily --refresh-universe --recheck

# 4. 관찰 로그 D+ 추적값 업데이트
rtk python -u scripts/run_observation_tracking_update.py

# 5. 일일 운영 요약 생성
rtk python -u scripts/run_daily_research_summary.py

# 6. 패턴 재분석을 별도 스냅샷으로 보관
rtk python -u scripts/run_signal_research_pipeline.py --mode backtest --snapshot-date YYYY-MM-DD

# 7. 기존 active 조건과 신규 스냅샷 조건 비교
rtk python -u scripts/review_new_snapshot_conditions.py --snapshot-date YYYY-MM-DD
```

active 조건을 바꿀 때만 아래 옵션을 명시적으로 사용한다.

```powershell
rtk python -u scripts/run_signal_research_pipeline.py --mode backtest --snapshot-date YYYY-MM-DD --promote-strategy
```

평소 일일 운영에서는 `--promote-strategy`를 사용하지 않는다.

## 운영 판단 기준

새 후보를 계속 추가하는 것은 좋지만, 범위를 무리하게 넓히면 신호 품질이 떨어진다. 따라서 기본 운영은 다음 순서를 따른다.

1. 최신 거래대금 상위 30개 안에서 찾는다.
2. 신규 기업은 자동으로 추가 대상에 올리되, 데이터 품질 확인 전에는 확정 신호로 쓰지 않는다.
3. 검증된 H01/H02/H03/H04/H06 조건만 먼저 적용한다.
4. 확정 후보만 관찰 로그에 넣는다.
5. 조건 추가는 관찰 샘플이 충분히 쌓인 뒤 진행한다.
6. 새 패턴은 발견 즉시 active에 넣지 않고 날짜 스냅샷과 신규 조건 검토 문서로 남긴다.
