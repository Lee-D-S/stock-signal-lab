# 프로젝트 현황 총정리

## 목적

이 프로젝트는 거래대금 상위 종목을 매일 갱신하고, 기업별 보고서에서 상승/하락 원인 이벤트를 구조화한 뒤, 공통 패턴을 조건화해 일별 후보를 찾고 관찰 로그로 성과를 추적하는 체계다.

핵심 흐름은 다음과 같다.

```text
거래대금 top30 갱신
-> 신규 기업 폴더/보고서 생성
-> 기업별 상승/하락 events 추출
-> events 공통점/패턴 분석
-> 조건 후보 생성 및 백테스트
-> active 조건으로 일별 후보 탐지
-> 수급 재조회로 확정
-> 관찰 로그에 추가
-> D+1/D+5/D+10/D+20 추적
```

## 전체 처리 흐름

### 1. 거래대금 상위 30개 유니버스 갱신

매일 최신 거래대금 기준으로 국내 주식 상위 30개를 산출한다.

구현 스크립트:

```powershell
rtk python -u scripts/run_daily_universe_refresh.py --top 30
```

주요 역할:

- 거래대금 상위 30개 산출
- 우선주, ETF/ETN, 스팩 등 제외
- 기존 기업 폴더 존재 여부 확인
- 신규 기업을 보고서 생성 대상으로 분류

주요 산출물:

- `07_전략신호/거래대금_상위_유니버스.csv`
- `07_전략신호/거래대금_상위_유니버스.md`
- `01_기획/신규_기업_추가_대상.md`

### 2. 신규 기업 보고서 생성

거래대금 top30에 새로 들어왔지만 `00_기업별분석/{기업명}` 폴더가 없거나 보고서가 누락된 기업은 보고서 생성 대상이 된다.

구현 스크립트:

```powershell
rtk python -u scripts/run_new_company_reports.py --include-existing-missing
```

주요 역할:

- 신규 기업 분석 폴더 생성
- 분기별 원인 후보 실제 분석 보고서 생성
- 분기별 `events.jsonl` 생성
- 상장일 확인
- 상장 전 기간 보고서 생성 방지

보고서 위치:

- `00_기업별분석/{기업명}/`

예시 파일:

- `{기업명}_2026_Q1_원인후보_실제분석.md`
- `{기업명}_2026_Q1_events.jsonl`

### 3. 보고서에서 events 추출

각 기업 보고서의 상승/하락 이벤트는 구조화된 events로 저장된다.

events 주요 필드:

| 필드 | 의미 |
| --- | --- |
| date | 이벤트 발생일 |
| ticker/name | 종목코드/종목명 |
| direction | 상승/하락 |
| chg_pct | 등락률 |
| amount_tag | 거래대금 강도 |
| flow_category | 수급 구조 |
| dart_tag | DART 공시 동반 여부 |
| window_category | 원인 창 유형 |
| market_regime | 시장 국면 |
| followup | 후속 흐름 |
| needs_external_review | 외부 자료 확인 필요 여부 |

전체 events 집계 산출물:

- `03_원천데이터/이벤트.csv`
- `03_원천데이터/이벤트_분포_요약.md`

### 4. events 기반 공통점/패턴 분석

모든 기업의 events를 합쳐 공통점과 패턴을 찾는다.

실행 명령:

```powershell
rtk python -u scripts/run_signal_research_pipeline.py --mode backtest --snapshot-date YYYY-MM-DD
```

분석 축:

- 방향: `up`, `down`
- 거래대금 강도: `거래대금약함`, `거래대금평균상회`, `거래대금급증`
- 수급 구조: `외국인기관동반매수`, `외국인기관동반매도`, `수급엇갈림`
- 공시 여부: `DART공시동반`, `주변공시부재`
- 원인 창 유형: `직접반응`, `선반영`, `누적배경`
- 시장 국면

패턴 후보 통과 기준:

- 산정 가능 이벤트 10회 이상
- 최빈 후속 흐름 비율 65% 이상
- 3개 이상 종목

주요 산출물:

- `04_패턴분석/패턴_분석_요약.md`
- `04_패턴분석/패턴_가설_후보.csv`
- `04_패턴분석/패턴_분석_전체.csv`
- `04_패턴분석/패턴_분석_시장국면별.csv`

### 5. 가설 검토와 백테스트

패턴 후보는 바로 전략으로 쓰지 않고, 가설 검토와 실전형 백테스트를 거친다.

- 진입 후보, 반등 관찰 후보, 추격매수 회피 후보를 구분

주요 산출물:

- `05_가설검토/가설_이벤트_검토.md`

### 6. 조건화

현재 실제 일별 후보 탐지에 사용하는 active 조건은 기존 검증 중인 조건만 사용한다.

active 조건 파일:

- `07_전략신호/전략_조건_초안.csv`
- `07_전략신호/전략_조건_초안.md`

현재 active 조건:

| 조건 | 용도 | 핵심 조건 |
| --- | --- | --- |
| H01 | 매수 후보 | 변동성 장세 + 상승 + 거래대금약함 + DART공시 + 외국인기관동반매수 |
| H02 | 반등 감시 후보 | 변동성 장세 + 하락 + 거래대금약함 + DART공시 + 외국인기관동반매수 |
| H03 | 단기 반등 감시 후보 | AI/전력기기 테마장 + 하락 + 거래대금평균상회 + 수급엇갈림 + 주변공시부재 |
| H04 | 참고 회피 후보 | 유동성/저금리 장세 + 상승 + 거래대금급증 + 외국인기관동반매도 + DART공시 |
| H06 | 추격매수 회피 후보 | 반도체 반등장 + 상승 + 거래대금급증 + 외국인기관동반매도 + DART공시 |

주의:

- 새 패턴 분석 결과의 `H01`, `H02`는 기존 active `H01`, `H02`와 같은 의미가 아니다.
- 새 패턴은 날짜를 붙인 스냅샷 조건으로만 관리한다.
- active 조건을 바꿀 때만 명시적으로 승격한다.

## 신규 조건 스냅샷 관리

패턴 재분석 결과는 기존 active 조건을 덮어쓰지 않는다. 날짜별 스냅샷으로 보관한다.

스냅샷 위치:

- `09_조건스냅샷/YYYY-MM-DD/`

2026-05-01 스냅샷 결과:

| 구분 | 개수 | 처리 |
| --- | ---: | --- |
| 발견된 조건 후보 | 9개 | 스냅샷 보관 |
| 기존 active와 동일한 조건 | 3개 | 기존 active ID 유지 |
| 기존 active에는 없던 신규 후보 | 6개 | 검증 대기 |
| 기존 active였지만 스냅샷에서 빠진 조건 | 2개 | 삭제하지 않고 보류 |

기존 active와 동일한 조건:

| 스냅샷 조건 | 기존 active 조건 |
| --- | --- |
| 2026-05-01-H03 | 기존 H04 |
| 2026-05-01-H06 | 기존 H01 |
| 2026-05-01-H08 | 기존 H02 |

신규 후보 6개 검토 결과:

| stable 후보 ID | 판단 | 비고 |
| --- | --- | --- |
| NEW-20260501-H05 | 우선 관찰 | 변동성 장세 상승, 거래대금약함, 외국인기관동반매수, 주변공시부재 |
| NEW-20260501-H07 | 우선 관찰 | AI/전력기기 테마장 하락, 거래대금약함, 외국인기관동반매수, 주변공시부재 |
| NEW-20260501-H01 | 소표본 보류 | 표본 부족 |
| NEW-20260501-H02 | 소표본 보류 | 표본 부족 |
| NEW-20260501-H04 | 보류 | 평균 점수 우위 약함 |
| NEW-20260501-H09 | 승격 제외 | 회피 조건 점수 부적합 |

운영 기준:

- active에 새로 추가한 조건은 현재 0개다.
- 우선 관찰 2개도 기존 `H01~H06` 번호 체계에 섞지 않는다.
- 신규 조건은 최소 20건 이상 관찰 표본이 쌓일 때까지 스냅샷 검토 대상으로 유지한다.
- 승격 시에는 기존 H 번호를 재사용하지 말고 별도 stable ID를 부여한다.

GitHub Actions 워크플로우가 구현되어 있다.

- `.github/workflows/daily_auto.yml`

자동 실행:

일일 후보 탐지 작업은 다음 순서로 실행된다.

1. `run_signal_research_pipeline.py --mode daily --refresh-universe --recheck`
2. `run_observation_tracking_update.py`
3. `run_daily_research_summary.py`
4. 변경된 `ai 주가 변동 원인 분석/` 파일 자동 commit/push

수동 실행도 가능하다.

- GitHub Actions의 `일별 자동 루틴` 실행
- `job=daily` 또는 `job=both`
- 필요하면 `date=YYYY-MM-DD` 입력

## 현재 결론

현재 프로젝트는 `기업별 보고서 -> events -> 패턴 분석 -> 조건화 -> 일별 후보 탐지 -> 수급 확정 -> 관찰 로그`까지 연결되어 있다.

다음 단계는 조건을 더 많이 만드는 것이 아니라, GitHub Actions가 매일 쌓는 관찰 결과를 기준으로 active 조건의 실제 성과 표본을 늘리는 것이다.
